# overview.md

splitwm is a from-scratch Wayland compositor written in Rust (smithay 0.7), implementing a terminal-multiplexer-style split layout on a scrollable canvas. It is the Wayland port of this project's original X11 window manager, which is retired to the `xorg` branch and remains the behavioral spec — `master`'s README lists every approved deviation from it.

The architecture separates pure layout math (`layout.rs`) from the layout's mutations and scroll bookkeeping (`state.rs`), compositor plumbing (`comp/`), and rendering (`render/`). The layout is a flat left-to-right list of *columns*; a column is one split or one vertical *stack* of splits — deeper nesting is unrepresentable, unlike the X11 predecessor's arbitrary n-ary split tree. Splits are still arena-keyed (`layout::NodeId`, a newtype over the arena index so it can never be swapped with a `Win` at a call site) rather than boxed pointers, but the arena backs a flat strip, not a tree: a column owns its width in pixels and stacked rows hold height fractions, so opening, closing, or resizing one column never resizes any other — the strip absorbs the difference and only a stack's own rows still trade height.

Chrome (borders, titlebars, buttons, taskbar, the pointer) is still palette-indexed pixel art, but composited differently than the X11 version's single full-screen underlay: each chrome element is its own GPU-uploaded piece (`comp::pieces`), re-rendered only when its content fingerprint changes, so scrolling and layout animation are pure element repositioning, not recomposites. Software rendering happens in `render/` (CPU buffers, palette-indexed) and reaches the GPU through `render::indexed`'s shared shader/staging pipeline. Static art (borders, buttons, titlebar frames) is baked into the binary at compile time by `build.rs` (quantized once, not decoded at startup); wallpapers and theme icons decode at runtime via `pixel_graphics::magick_decode_rgba` (an ImageMagick shell-out — this crate has no Rust PNG decoder of its own). App icons still go through a one-shot OKLCH hue-rotation pipeline (`oklch.rs`) to disambiguate same-app windows.

The event loop is `calloop`, single-threaded, with frame-paced rendering: vblank-driven on the tty (DRM) backend, a 60 Hz timer standing in on winit (nested) and headless. The notification daemon (`notify.rs`) runs org.freedesktop.Notifications over `zbus`, bridged to calloop through a channel rather than a thread the main loop blocks on or polls. A `SPLITWM_HEADLESS=1` backend (`backend::headless`) renders offscreen for the test harness, driven by a line protocol on stdin (`SPLITWM_DEBUG_CHANNEL=1`, see `comp::debug`) — this is how `cargo test`'s socket tests and `drive.sh`'s screenshot walk exercise the compositor without anything appearing on a real display.

A design invariant carried over from the X11 version: state that must stay internally consistent is kept private with mutation/read methods that enforce the invariant, rather than trusted to callers. `state::State`'s `focused_leaf` is private — reads go through `focused_leaf_valid` (a later mutation can remove the focused leaf, so reads re-validate rather than trusting a stale id) and writes go through `focus_leaf` (accepts only live leaf ids), so a dangling focus is never handed out. The same pattern covers scroll (`scroll_x`/`scroll_target` glide toward each other via `step_scroll`, private so every mutation goes through clamping) and, in `comp::mod`, the focused float and fullscreen client records (`WindowRoles`, re-validated against the window store on every read). Windows and splits are paired for life — a new window opens in (or as) its own split, and a dying window takes its split with it — so a `Leaf` with a client is never minimized (`Leaf::show` clears the flag whenever it sets a client) and an empty leaf is the only way a split exists without a window.
